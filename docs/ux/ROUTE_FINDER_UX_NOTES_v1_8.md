# 길찾기 UX 노트·검토 사항 (v1.8)

## 1. 임시 셔틀 포함 (FAQ)

### 1.1 "임시 셔틀 포함"이란?

- **정의:** `shuttle_temp_route` / `shuttle_temp_route_stop` 테이블에 등록된 파싱·검수 완료된 임시 셔틀 노선을 의미합니다.
- **출처:** PDF 업로드 → 파싱 → 검수(promote) 후 `shuttle_temp_*`에 반영된 노선입니다.
- **예:** 파업/행사 시 운영하는 대체 셔틀 버스 노선.

### 1.2 임시 셔틀이 많을 때

- **현재:** 검색 시 `include_shuttle=1`이면 출발-도착 구간을 지나는 모든 서울 버스 + 임시 셔틀을 함께 조회합니다.
- **결과 제한:** `route_finder_search`는 bus·shuttle_temp 각각 최대 10건씩 반환합니다 (총 20건 내).
- **과다 시:** 20건 이상이면 정렬 기준(정류장 수 적은 순)으로 상위만 표시됩니다. UX 개선 제안은 아래 1.3 참고.

### 1.3 제안: "경로 찾은 후 임시 셔틀 추가" 플로우

| 현재 방식 | 제안 방식 |
|----------|----------|
| 검색 전에 체크박스로 임시 셔틀 포함 여부 결정 | 1) 기본 검색: 서울 버스만 | 2) 경로 없음/부족 시 "임시 셔틀 추천" 버튼 |
| 결과가 많으면 한 번에 쏟아짐 | 경로 결과에서 "임시 셔틀 포함해서 다시 보기" 옵션 |

**장점:**  
- 사용자가 먼저 일반 버스 경로를 확인한 뒤, 필요 시 임시 셔틀을 추가로 볼 수 있음.  
- 노선이 많을 때 혼란 감소.

**현재 구현 상태:**  
- 경로 없음 시 "임시 셔틀 추천" 버튼(`include_shuttle=1`로 재검색)이 이미 있음.  
- 검색 전 체크박스는 유지하되, 툴팁/설명으로 "임시 셔틀 포함" 의미를 명확히 하는 것이 좋음.

## 2. 경로만 보이는 것 vs 지도

### 2.1 현재 선택

- **경로만 표시:** 목록 형태(노선명, 정류장 요약, 예상 시간)로만 제공합니다.
- **이유:** MVP 단계에서 지도 API 연동·비용·성능 이슈를 최소화하기 위함.

### 2.2 지도 표시 가능 여부

- **기술적으로 가능:** Kakao/Naver 지도 API, Leaflet+OpenStreetMap 등으로 지도 표시 가능.
- **필요 사항:**  
  - API 키·라이선스  
  - 경로 폴리라인 좌표(정류장 위경도)  
  - 모바일/데스크톱 반응형 레이아웃  
- **권장:** v1.8 이후 지도 기능을 Phase 2로 검토. `PLAN_UX_OPERATIONS_ROUTE_FINDER_v1_7.md` Part 3에서도 "지도 폴리라인 + 정류장 마커"를 향후 방향으로 명시하고 있음.
